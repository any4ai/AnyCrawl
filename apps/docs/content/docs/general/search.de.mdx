## EinfГјhrung

Die AnyCrawl-Search-API liefert programmgesteuert Suchergebnisse mehrerer Suchmaschinen. FГјr LLMs aufbereitet: saubere, strukturierte Daten.

Die API **antwortet sofort und synchron** вЂ“ kein Polling, keine Webhooks. **Mehrere Engines** und **mehrere Ergebnisseiten** pro Anfrage sind mГ¶glich.

### Kernfunktionen

- **Mehrere Engines**: derzeit Google, weitere geplant
- **LLM-optimiert**: strukturierte JSON-Daten
- **Mehrsprachig**: Sprachen und Regionen wГ¤hlbar
- **Mehrseitige Ergebnisse**: mehrere SERP-Seiten in einer Anfrage
- **Performance**: **Hohe ParallelitГ¤t** mit asynchroner Verarbeitung
- **Sofortantwort**: **Synchrones API**
- **Umfassend**: Suchtreffer und SuchvorschlГ¤ge

## Best Practices

### Suchmaschine

1. **Allgemein** в†’ `google` (derzeit einzige Option)
2. **Mehrsprachig** в†’ passendes `lang`
3. **Mehr Treffer** в†’ `pages` erhГ¶hen

### Performance

- **`pages`**: mehrere SERP-Seiten in einer Anfrage statt vieler Einzelaufrufe
- **Sprache**: `lang` auf Zielregion abstimmen
- **Anfrage**: prГ¤zise Keywords

### Fehlerbehandlung

```javascript
try {
    const response = await fetch("/v1/search", {
        method: "POST",
        headers: {
            Authorization: "Bearer YOUR_API_KEY",
            "Content-Type": "application/json",
        },
        body: JSON.stringify({
            query: "artificial intelligence",
            pages: 2,
        }),
    });

    const result = await response.json();

    if (result.success) {
        // Handle successful result
        console.log("Found results:", result.data.length);

        // Separate search results from suggestions
        const searchResults = result.data.filter((item) => item.url);
        const suggestions = result.data.filter((item) => !item.url);

        console.log("Search Results:", searchResults.length);
        console.log("Suggestions:", suggestions.length);
    } else {
        // Handle search failure
        console.error("Search failed:", result.error);
    }
} catch (error) {
    // Handle network error
    console.error("Request failed:", error);
}
```

## Hohe ParallelitГ¤t

Die API **unterstГјtzt hohe ParallelitГ¤t**. Viele gleichzeitige Suchanfragen sind unkritisch:


## Limits und Credits

- **Kein Ratenlimit** bei ParallelitГ¤t вЂ” die API unterstГјtzt hohe ParallelitГ¤t nativ.
- Verbrauch Гјber Response-Header und `BillingChargeDetails` prГјfen.

### Credit-Modell (Suche + optionale Follow-up-Scrapes)

1. **`search_pages`** вЂ” entspricht **`pages`** (Standard oft 1 Credit pro Seite).
2. **`template_per_call`** вЂ” bei top-level **`template_id`** (Search-Template): einmal pro Request `pricing.perCall`.
3. Mit **`scrape_options`** und erfolgreichen Follow-ups:
   - **`search_result_scrape`** вЂ” `completed_scrape_count Г—` Engine-Einheit (`calculateScrapeCredits`).
   - **`search_result_scrape_template`** вЂ” bei **`scrape_options.template_id`**: pro erfolgreichem Follow-up die Scrape-Template-`pricing.perCall`. **Nur Search-Templates:** `metadata.charge_scrape_template_credits: false` deaktiviert diese Zeile (Engine-Zeile bleibt).

VollstГ¤ndige Spezifikation: Repository **`docs/api/search-scrape-credits-and-templates.md`**.

## FAQ

### Wann welche Spracheinstellung?

- **`en`**: globale/internationale Treffer
- **`zh`**: chinesischsprachige Inhalte
- **Regional** (`en-GB`, `fr-CA`, вЂ¦): lokalisierte SERPs

### Warum weniger Treffer als erwartet?

- Sehr spezifische oder seltene Begriffe
- `lang` passt nicht zur Anfrage
- Themengebiet-/Regionslimits der Suchmaschine
- Netzwerkprobleme

### Regionsspezifische Suche?

Die API wГ¤hlt anhand von `lang` passende Einstellungen. Sprachcodes zur Zielregion nutzen (z.вЂЇB. `en-GB` fГјr UK); ggf. regionale Begriffe in `query`.

### Unterschied Treffer vs. VorschlГ¤ge?

- **Treffer**: Titel, URL, Beschreibung, Quelle
- **VorschlГ¤ge**: verwandte Queries oft ohne URL

### ParallelitГ¤t limitiert?

Nein вЂ“ **hohe ParallelitГ¤t** ist nativ; Antworten kommen direkt.
